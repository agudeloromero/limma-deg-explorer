Methods

Purpose of the worked example

The manuscript is a Ten Simple Rules paper, not a software benchmarking study. The R Shiny and Python Shiny applications are provided as worked examples to illustrate transferable design principles for turning a bioinformatics analysis into an interactive web application.

The worked example used throughout the manuscript is a differential gene expression and Gene Ontology enrichment application that allows users to upload a results table, explore thresholds interactively, visualise outputs, and download a report capturing the current analysis state.

Validation dataset

The worked example uses GSE60450, a public mouse mammary gland RNA-seq dataset comparing luminal and basal cell populations across developmental stages.

This dataset was chosen because it is public, citable, and suitable for demonstrating a complete interactive workflow from differential expression results to downstream interpretation.

Differential expression workflow

R implementation

The R implementation follows a Bioconductor-oriented workflow. Counts are normalised and analysed using a limma-based pipeline, and the resulting table is used as input for the interactive application.

This implementation serves as the reference implementation for the genomics workflow used in the manuscript.

Python implementation

The Python implementation is included to demonstrate the same application-design principles in a Python Shiny environment. It supports the same general interactive workflow, but some analytical steps differ because the surrounding Python ecosystem does not fully mirror the Bioconductor stack used in R.

For this reason, the Python implementation should be interpreted as an illustrative parallel implementation rather than as a claim of strict methodological equivalence.

Gene Ontology enrichment

R implementation

The R implementation uses Bioconductor-oriented annotation and enrichment tooling and is suitable for environments where local annotation resources and offline operation are important.

Python implementation

The Python implementation uses Python-native enrichment tooling and external services where needed. This makes deployment straightforward in some settings, but can introduce internet dependence for parts of the workflow.

Why both implementations are shown

The manuscript does not argue that one framework is universally superior. Instead, it uses both implementations to show that the same design principles can be applied across different ecosystems.

The practical choice between frameworks depends on the location of critical dependencies, the surrounding analytical environment, deployment constraints, and the needs of the intended users.

Main practical differences

For the worked example in the manuscript, the most relevant differences are:

  • bioinformatics ecosystem maturity
  • offline versus internet-dependent enrichment
  • report-generation workflow
  • deployment and containerisation options

These trade-offs are summarised in Table 1 of the manuscript.
